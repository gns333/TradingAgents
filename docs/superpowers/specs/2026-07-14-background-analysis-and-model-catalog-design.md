# 后台分析、报告隔离与模型目录设计

## 背景

当前 Web 分析通过 `/api/events` 的 SSE 请求同步执行。浏览器页面既是任务发起方，也是事件流的唯一消费者；关闭页面后，服务端执行是否继续取决于连接生命周期，重新打开页面也无法恢复运行状态。报告只在 SSE 收到完成事件后保存，并且报告接口没有按用户隔离。后台模型配置则在前端写死供应商模型名称，容易随着厂商发布和下线模型而过期。

本次改造解决以下问题：

1. 分析在服务端后台执行，不依赖浏览器页面持续打开。
2. 每个普通用户最多有一个排队中或运行中的分析任务。
3. 普通用户只能查看自己的历史报告，管理员可以查看所有用户的报告。
4. 报告标题包含股票中文名和证券代码，所有报告内容均按 Markdown 展示。
5. 管理员从供应商官方接口获取可用模型名称，不再维护写死的模型列表。
6. 快速模型和深度模型继续保留两个独立选项，并允许选择同一个模型。
7. 管理员登录弹窗中的空错误框默认不显示。

## 已确认的产品决策

- 任务并发限制按用户计算，不是全站单任务。
- 不同用户可以同时提交任务；服务端通过有限 Worker 并发控制实际执行数量。
- 排队中的任务也算该用户的活动任务，因此该用户不能再次提交。
- 普通用户的报告中心相互隔离。
- 管理员可以查看所有用户的任务和报告，并看到用户 UID、邮箱等可用身份信息。
- 报告不强制统一章节、标题或顺序，只要求产生并渲染合法 Markdown。
- 不增加专门用于报告排版的 LLM 调用。
- 快速模型和深度模型的现有 Agent 角色分配保持不变。
- 两个模型选择框使用同一份实时模型目录，不提供“统一模型”开关。
- API Key 继续由应用加密后直接保存到数据库，不使用环境变量或云凭据管理服务。

## 方案选择

### 方案一：请求内执行

继续在 SSE 请求中执行 LangGraph，只增加前端重连。这不能保证浏览器断开后任务继续，也不能可靠恢复状态，因此不采用。

### 方案二：持久化任务表与应用内 Worker

任务和事件写入数据库，由应用生命周期内的后台 Worker 消费。浏览器只创建任务和读取事件。该方案可以复用当前 FastAPI 与 SQLite 架构，改造范围可控，并能满足页面关闭后继续分析、重新打开恢复状态的要求。

本次采用此方案。存储边界保持独立，使未来迁移到 CloudBase 共享数据库和独立 Worker 时不需要修改前端协议或分析执行逻辑。

### 方案三：立即接入云数据库与外部任务队列

该方案最适合多实例生产部署，但会同时引入 CloudBase 资源配置、远程数据库迁移、队列运维和本地开发模拟，不属于当前功能闭环。本次只为其保留接口边界，不直接实现。

## 架构

### 身份模型

新增统一的 `Principal` 身份对象，由现有请求头、访问邮箱和管理员会话解析：

- CloudBase 用户优先使用 `uid:<cloudbase_uid>` 作为稳定 `owner_key`。
- 没有 UID 时使用小写标准化后的 `email:<email>`。
- 本地管理员使用固定身份 `admin:local`。
- 同时保存可用的 UID 和邮箱作为展示字段，但权限判断只使用 `owner_key`。

普通用户在没有可验证 UID 或邮箱时不能创建任务、查看报告或读取任务事件。管理员会话继续绕过白名单限制，但管理员读取普通用户数据时不会改变数据归属。

### 任务存储

新增 `analysis_runs` 表：

| 字段 | 说明 |
| --- | --- |
| `id` | UUID 任务标识 |
| `owner_key` | 权限和并发控制使用的稳定身份键 |
| `owner_uid` | CloudBase UID，可为空 |
| `owner_email` | 用户邮箱，可为空 |
| `ticker` | 标准证券代码 |
| `stock_name` | 创建任务时解析出的中文名称 |
| `trade_date` | 分析日期 |
| `asset_type` | 资产类型 |
| `analysts` | JSON 分析模块列表 |
| `status` | `queued`、`running`、`completed`、`failed` |
| `current_agent` | 当前 Agent 展示名称 |
| `last_event_seq` | 最后持久化事件序号 |
| `error_type` / `error_message` | 失败详情 |
| `report_id` | 完成后关联的报告 ID |
| `created_at` / `started_at` / `finished_at` | 生命周期时间 |
| `heartbeat_at` | Worker 活跃时间 |

新增 `analysis_run_events` 表：

- 每条事件包含 `run_id`、单调递增的 `seq`、事件类型、JSON 数据和创建时间。
- 事件类型复用当前 `run_started`、`tool_called`、`agent_message`、`report_section_updated`、`run_completed`、`run_failed`。
- `(run_id, seq)` 唯一，前端可使用 `after` 参数增量读取。

创建任务时使用数据库事务检查同一 `owner_key` 是否已有 `queued` 或 `running` 任务。SQLite 实现同时增加活动状态的部分唯一索引，防止并发请求绕过检查。

### 后台 Worker

应用启动时创建一个专用任务调度器和有限大小的线程池：

- 默认最多同时执行两个分析任务。
- 调度器从数据库领取 `queued` 任务并原子更新为 `running`。
- Worker 调用现有 `stream_analysis_events()`，将每个事件先持久化，再通知实时订阅者。
- `report_section_updated` 同步更新任务当前进度；`run_completed` 保存报告并关联 `report_id`。
- SSE 客户端断开只停止当前订阅，不取消 Worker。
- Worker 捕获的异常统一写入 `run_failed` 事件并将任务设为 `failed`。

服务进程重启时，启动恢复逻辑将遗留的 `running` 任务标记为失败，错误类型为 `WorkerInterrupted`。本次不自动重跑已经开始的任务，避免重复模型费用和不完整检查点导致的重复副作用。仍处于 `queued` 的任务可以继续被领取。

该保证的边界是“关闭或刷新浏览器不影响任务”。它不承诺应用容器、主机或数据库故障后无损续跑。

### API

新增以下接口：

- `POST /api/runs`：验证白名单和身份后创建任务；已有活动任务时返回 `409`，并附带活动任务摘要。
- `GET /api/runs/active`：返回当前用户的活动任务；管理员可使用显式 owner 过滤查看。
- `GET /api/runs/{run_id}`：返回有权限访问的任务详情。
- `GET /api/runs/{run_id}/events?after=<seq>`：增量获取持久化事件。
- `GET /api/runs/{run_id}/stream?after=<seq>`：订阅实时事件，并从数据库补齐断线期间的事件。

现有 `/api/events` 不再作为前端主入口。为避免突然破坏外部调用，它保留兼容响应，但不再用于工作台新流程，并在后续版本删除。

所有任务详情、事件和报告接口都执行对象级权限检查：普通用户必须匹配 `owner_key`，管理员可以跨用户访问。

### 前端任务恢复

分析工作台初始化时：

1. 请求 `/api/runs/active`。
2. 若存在活动任务，恢复股票信息、状态、当前 Agent 和已持久化事件。
3. 使用最后一个事件序号建立 SSE 订阅。
4. 禁用“开始分析”，直到当前任务进入完成或失败状态。
5. SSE 暂时断开时保留任务状态并自动重连，不再直接把任务标记为失败。

创建任务改为先 `POST /api/runs`，成功后再订阅对应 `run_id`。如果服务端返回 `409`，前端直接恢复已有活动任务，而不是只显示错误。

## 报告存储与权限

`analysis_reports` 增加：

- `run_id`
- `stock_name`
- `owner_key`
- `owner_uid`
- `owner_email`

新报告由 Worker 在任务完成时保存，避免报告保存依赖前端收到完成事件。

报告列表规则：

- 普通用户的列表查询自动限定为自己的 `owner_key`。
- 普通用户不能通过猜测报告 ID 读取或删除其他用户报告。
- 管理员可以查询全部报告，并按用户、股票代码和日期过滤。
- 管理员报告列表展示用户邮箱；没有邮箱时展示 UID。
- 迁移前没有归属信息的历史报告只对管理员可见，避免错误地分配给普通用户。

报告标题优先显示 `股票中文名（代码）`。任务创建时使用本地股票目录解析并保存中文名；历史记录没有名称时只在读取阶段尝试本地目录补全，不影响原始报告正文。

## Markdown 报告

所有 Agent 报告继续保留各自最适合的内容结构，不新增统一章节模板。

- 自由文本 Agent 的提示词明确要求输出 Markdown，不限定标题名称和顺序。
- 已使用 Pydantic 结构化输出的 Agent 继续通过确定性渲染器生成 Markdown。
- 前端 Markdown 渲染器负责标题、粗体、列表、表格、引用、行内代码和代码块。
- LLM 输出中的 HTML 按普通文本处理，不直接插入可执行 HTML。
- 各子报告继续使用独立 Tab 展示。

本次只统一载体和渲染行为，不重写报告观点，不补造缺失章节，也不为了排版增加 LLM 调用。

## 快速模型与深度模型

保留当前固定角色分配：

- 快速模型：市场、新闻、基本面、社交情绪分析师，看多/看空研究员，交易员，激进/中性/保守风险分析师，以及历史决策复盘。
- 深度模型：研究经理和组合经理。

该规则不根据上下文长度或运行时难度动态切换，也不增加快速模型失败后自动改用深度模型的降级行为。

后台仍显示两个选择框，但它们读取同一份供应商模型目录。数据库继续保存 `quick_model` 和 `deep_model`，允许两个字段取相同值，不增加相等性限制或额外开关。

## 动态模型目录

新增后端模型目录服务。浏览器不直接请求第三方厂商，避免 API Key 暴露和跨域问题。

支持两种调用方式：

1. 新建配置时，前端通过 `POST` 请求把供应商、Base URL 和尚未保存的 API Key 交给后端；后端仅用于本次模型列表请求。
2. 编辑已保存配置时，前端传配置 ID；后端解密数据库中的 API Key 后请求厂商。

后端将不同响应统一为：

```json
{
  "models": [
    {"id": "model-id", "display_name": "Model name"}
  ],
  "source": "provider_api",
  "fetched_at": "ISO-8601 timestamp"
}
```

供应商适配：

- DeepSeek：`GET {base_url}/models`，Bearer Token。
- OpenAI：`GET https://api.openai.com/v1/models`，Bearer Token。
- Anthropic：`GET https://api.anthropic.com/v1/models`，`x-api-key` 与 `anthropic-version` 请求头。
- Google Gemini：`GET https://generativelanguage.googleapis.com/v1beta/models`，过滤支持 `generateContent` 的模型。
- Kimi：`GET https://api.moonshot.cn/v1/models`，Bearer Token。
- OpenAI 兼容服务：`GET {base_url}/models`，Bearer Token。

阿里云百炼公开的“可部署模型”接口不是当前账号所有可直接调用的 Serverless 对话模型列表，因此不把它伪装成完整实时目录。智谱官方文档中也没有确认与当前调用协议匹配的模型列表接口。对于无法可靠列举的供应商：

- 允许管理员手动输入模型 ID。
- 保存前可使用最小请求做连通性验证。
- 不回退到前端写死模型名称。

模型目录请求使用短超时并缓存五分钟。缓存按供应商、Base URL 和 API Key 指纹隔离，只缓存模型元数据，不缓存明文 API Key。请求失败时返回结构化错误，前端保留已经选择或手动输入的模型 ID，不静默替换。

## 管理员登录提示框

管理员登录弹窗中的 `.form-status` 是错误提示区域。当前空内容时仍占据空间并显示红色边框。本次改为：

- 空内容或没有错误状态时隐藏。
- 只有登录、首次设置或密码校验失败时显示。
- 再次提交前清空并隐藏旧错误。
- 保留 `role="alert"`，使辅助技术能够读取错误。

## 错误处理

- 重复创建任务：`409 ActiveRunExists`，返回已有任务 ID 和状态。
- 身份缺失：`401 IdentityRequired`。
- 不在白名单：`403 AccessDenied`。
- 越权读取任务或报告：统一返回 `404`，避免泄露对象是否存在。
- Worker 异常：任务进入 `failed`，保存真实异常类型和经过清理的消息。
- SSE 断开：前端重连并使用 `after` 补事件，不改变服务端任务状态。
- 模型目录认证失败、限流、超时和响应格式异常分别返回可辨识错误类型。
- 报告 Markdown 渲染失败时展示转义后的原文，不丢失报告内容。

## 数据迁移

数据库初始化继续采用当前的增量建表/加列模式：

- 创建任务和事件表及索引。
- 为报告表增加名称、归属和任务关联字段。
- 不删除或覆盖已有报告。
- 已有 `owner` 字段保留用于兼容读取，新权限逻辑只信任新的 `owner_key`。
- 历史无归属报告仅管理员可见。

## 验证策略

按照仓库规则采用风险分级验证：

- 存储层：测试每用户活动任务唯一约束、事件顺序、状态转换、报告归属和历史迁移。
- API：测试白名单、管理员越权能力、普通用户隔离、重复任务 `409`、事件断点续读和模型目录鉴权。
- Worker：使用伪造事件流验证页面断开不影响执行、完成后保存报告、异常后状态落库、重启后遗留运行任务标记失败。
- Markdown：测试各报告正文原样保存、常用 Markdown 元素正确渲染、HTML 被转义。
- 前端：测试初始化恢复活动任务、运行期间禁用提交、SSE 重连、中文股票标题、管理员用户列和空登录错误框隐藏。
- 集成检查点：完成后运行 Web、报告和后台管理相关测试；提交前再执行一次合并后的相关回归测试，不在每个小步骤重复运行全量测试。

## 非目标

- 本次不实现 CloudBase 共享数据库、外部消息队列或多实例 Worker 选主。
- 本次不保证服务进程或主机重启后从 LangGraph 中间节点无损续跑。
- 本次不增加取消任务、任务优先级或管理员调整全站并发数。
- 本次不改变 Agent 使用快速/深度模型的角色映射。
- 本次不统一报告章节，不使用 LLM 二次改写报告。
- 本次不从非官方网页抓取模型名称。
